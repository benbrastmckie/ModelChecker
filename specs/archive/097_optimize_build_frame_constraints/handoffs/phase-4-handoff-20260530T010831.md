# Phase 4 Handoff: Solver Configuration Guards

**Phase**: 4 - Solver Configuration Guards
**Status**: COMPLETED
**Timestamp**: 2026-05-29T02:30Z

## What Was Done

### Applied (no regressions):
- `max_memory`: 4096 MB per-solver memory limit added to `_configure_quantifier_mode()`.
  This prevents OOM kills by causing Z3 to return 'unknown' gracefully when memory
  exceeds 4 GB, rather than being killed by the OS (earlyoom). Direct mitigation for
  the OOM kills described in task 98.

### Reverted (caused regressions):
1. **memory_max_size=4096**: Invalid parameter name. Caused "unknown parameter" exception
   which propagated as failures for all 43 tests. The correct per-solver parameter is
   `max_memory`, NOT `memory_max_size`.

2. **smt.qi.max_instances=100000**: Caused consistent failures on BM_CM_2 and BM_CM_4.
   These countermodel examples (SAT) apparently require >100000 quantifier instantiations
   to build the countermodel. Setting this cap introduces unsound 'unknown' returns.
   Reverted. A note was added to the docstring documenting why it was not applied.

## Deviations from Plan

- **qi.max_instances**: SKIPPED (caused 2 regressions). The plan suggested starting with
  10000, but even 100000 was insufficient. Without per-example profiling of instantiation
  counts, any cap risks causing incorrect 'unknown' results for valid examples.
- **memory_max_size**: Corrected to `max_memory` (the actual Z3 per-solver parameter name).

## Final State: 43/43 tests passing
